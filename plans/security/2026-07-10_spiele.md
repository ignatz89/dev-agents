# Security Review: spiele

Öffentlich erreichbarer Realtime-Spiel-Server (aiohttp/WebSocket) auf dem Hetzner-Server.
Kein Login, Räume per Code. Es geht um ein Familienspiel — Anspruch: solide Grundhärtung,
keine Über-Absicherung.

## Assets
- Verfügbarkeit des Servers; der Server-Host selbst (der Prozess darf kein Einfallstor sein).
- Keine Secrets, keine persönlichen Daten, keine Datei-Schreibzugriffe.

## Attack Surface
- Offener HTTP/WS-Port 8090 im Internet · beliebige WS-Nachrichten · Raum-Beitritt per Code.

## Risiken

### 🔴 Kritisch (sofort)
- **Dienst lief als root** (internetseitig). *Angriff:* eine Lücke im Prozess = root-Zugriff auf den Host.
  *Gegenmaßnahme (Schichten):* `User=nobody` + systemd-Härtung (`NoNewPrivileges`, `ProtectSystem=full`,
  `ProtectHome`, `PrivateTmp`). **→ behoben.**
  *Prüfbefehl:* `systemctl show spiele -p User` → `User=nobody`; `ps -o user= -C python | grep -v root`.
  Status: **[VERIFIZIERT nach Deploy]**

### 🟡 Mittel (bald)
- **Keine Größen-/Mengen-Limits → DoS/Memory.** *Angriff:* riesige Nachrichten oder unendlich viele
  Räume/Verbindungen erschöpfen RAM. *Gegenmaßnahme:* `max_msg_size=256KB`, `MAX_ROOMS=300`,
  `MAX_MEMBERS=8`. **→ behoben.** *Prüfbefehl:* im Code gesetzt; Verhalten: >8. Beitritt → „Raum ist voll".
  Status: **[VERIFIZIERT im Code]**
- **Kein TLS (http/ws unverschlüsselt).** Für ein Brettspiel gering (keine sensiblen Daten), aber
  Standard wäre TLS. *Gegenmaßnahme:* Domain + Reverse-Proxy oder Cloudflare-Tunnel (wie pi-nas).
  Status: **[UNGEPRÜFT / offen — bewusst zurückgestellt]**
- **Schwache Zugangskontrolle.** Jeder mit IP:Port + Raum-Code kann beitreten; Codes sind kurz
  (Wort + 2 Ziffern). *Restrisiko:* ein Fremder rät einen aktiven Code und stört. Für den Familien-Einsatz
  akzeptabel. *Option:* längere Zufalls-Codes, falls je nötig. Status: **[offen, Restrisiko akzeptiert]**
- **Kein scharfes Rate-Limiting.** Bewusst nicht — Zeichnen sendet legitim viele Nachrichten/Sek.
  Message-Size- + Raum-Limits mildern. Echtes Rate-Limiting am besten via Reverse-Proxy. Status: **[offen]**

### 🟢 Gering / geprüft-ok
- **XSS:** Namen/Chat werden per `textContent`/`escapeHtml` gerendert (kein `innerHTML` mit
  ungeprüften Server-Daten). Geprüft — ok.
- **Path-Traversal:** `game_page` prüft `".."` und Existenz. Ok.
- **Firewall:** nur 8090/tcp offen (ufw aktiv).

## Empfehlungen (priorisiert)
1. ✅ non-root + systemd-Härtung (erledigt) — größter Hebel.
2. ✅ Größen-/Mengen-Limits (erledigt).
3. TLS + Reverse-Proxy/Tunnel, falls es dauerhaft/öffentlicher läuft (mittlerer Aufwand).
4. Bei Missbrauch: längere Raum-Codes + Rate-Limit am Proxy.

## Gut gemacht
- Keine Secrets im Repo · Prozess schreibt nichts auf die Platte · WS-Heartbeat · sichere Ausgabe (kein eval/innerHTML).

## Defense in Depth (Ist-Stand)
non-root + systemd-Sandbox **und** Server-Limits **und** Firewall = mehrere Schichten. Nächste Schicht: TLS/Reverse-Proxy.
